Overview

This week your goal is to implement a Euclidean Vector. For those that don't remember linear algebra a Euclidian vector is a mathematical vector as opposed to the std::vector<T> which you have become so familiar with over the last couple of weeks.

What is the Façade Design Pattern

The Façade Design Pattern aims to make a software library easier to use, understand and test. This can be achieved by taking a complex application and creating a simple interface.

Façade is similar to the Adaptor pattern, however, rather than wrapping an existing class, as we did when creating the StringStackQueue, a façade will wrap an entire subsystem composed of a number of classes.

The Mathematical Vector class

This weeks tutorial aims to build a simple mathematical vector. This vector will have a number of methods which represent vector arithmetic, such as addition and subtraction of vector, the dot product, calculating the length of the vector, etc. For a complete breakdown of all the methods you will be implementing this week download the QT project here.

The vector api includes a number of classes including the EuclideanVector class along with a set of exception classes which you will build later. All these classes are self-contained within the EuclideanVector class which is the only header file which needs to be included to use the entire API and therefore we can call this our Façade.

Does this make it easier to use through?

One of the key components that a Façade must ensure is that the interface is simpler than the API itself. This is achieved with this interface as calling the method double dotProduct(const EuclideanVector &vector) const; is surely easier than computing the dot product from scratch. Further, by giving a method which computes this result, the Façade does not expose the underlying storage structure of the EuclideanVector.

Exceptions

In today's lab you may be writing your first C++ exceptions and possibly your first ever exceptions. This should be your first task before starting on the actual EuclideanVector class. This project includes four exceptions, one of which has already been implemented for you. The exceptions included are:

  • VectorRuntimeException: the exception class which has already been implemented for you.
  • VectorArithmeticException: an exception which is thrown when an exception arises during arithmetic operations.
  • VectorAssignmentException: an exception which is thrown when an exception arises during assignment operations.
  • VectorIndexOutOfBoundsException: an exeception which is thrown when an exeception arises which accessing an element which is outside the range of the vector.

All three exception classes which have not been implemented should inherit VectorRuntimeException. These classes require no additional methods to be written so feel free to implement them in the header file rather than making a new implementation file.

Note

You may ask why do these exceptions not extend from a base exception class like in Java. This is due to the std::exception class only being a relatively recent addition to C++. As a result in C++ you can throw anything, even a double.

Creating the vector

All the methods required for your mathematical vector are included in the provided EuclideanVector header file. You may of course add any additional methods, and for this exercise it is recommended.

You also need to select the underlying data structure that you will use for your EuclideanVector. We strongly suggest using a double* to give yourself some extra practice with pointers in preparation for next week's lab.

Hint

If you use a double* and construct it using new[] you must deallocate the memory using delete[] in your destructor, not just delete.

If you get stuck at any point in terms of what each method should do than refer back to the Euclidean Vector wiki page linked to at the start of this tutorial.

Testing your vector

To validate that you have implemented your EuclideanVector Façade correctly try compiling and running the provided main.cpp. Before you start these tests ensure that you have, at the very least, constructed each method from your header file in the implementation file, and that you have written the code for all three exception classes. Without the minimum you will face a significant number of compilation errors.
