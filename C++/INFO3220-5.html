Overview

This week your goal is to become familiar with the Builder Design Pattern. To assist with this you will be provided with an existing code base which you will extend. By the end of this tutorial this code base will contain a working implementation of the Builder Design Pattern.

The Builder design pattern

The Builder design pattern is one of five Creational Design Patterns we will consider within this course and the second that has been considered in detail in the tutorials.

The Builder design pattern aims to seperate the construction of a complex object or subsystem from its composition, so that the same method for construction can create many different representations. It consists of three components (also represented below in UML):

  • The Director is the initiator of the building process and takes ownership of the Product once it is constructed.
  • The Builder is the class which defines the algorithm (recipe) for how to build a specific instance of a Product. Each Builder should inherit from a common base class which defines the a standard interface for the construction of a Product.
  • The Product is the class which is created using a Builder. This is often a complex object or subsystem constructed using a number of Parts. To conform with the Builder design paradigm these Parts should be able to be assembled in a number of different permutations to produce various Products.
Fig.1 - Adapted from Design Patterns: Elements of Reuseable Object-Orientated Software.

A builder design pattern should be used when:

  • the algorithm (recipe) for how to build a specific object should be independent of the parts that make up the object.
  • the construction process must allow different representations of the object to be constructed.

In today's tutorial you will be working with a code base where both of these features are desirable.

Creating a Computer Class (the Product)

Before you can begin implementing concrete Builder classes you must ensure that a valid Product class exists; in this context a Computer class. The Qt project which you will be working with this week contains a Computer class which is made up of a set of Parts which all inherit from a common Part base class as defined in the following diagram.


There are two types of Parts. Those that are required: components which all computers have e.g. a CPU and components that may be added for additional benefit such as water cooling.

To begin, download the Qt project here. A good first step before you get started is to look over the provided source code to become familar with, in particular ,the Part class and all classes which inherit from it. This includes the HardDriveSet, RamSet and GraphicsCardSet wrapper classes.

Implementing the CPU class

While most of the Parts have already been implemented, the CPU implementation file has not yet been implemented. The CPU class is similar to many of the previously implemented classes. One difference that you will need to consider, however, is the requirement that the output from the getPartInformation() method requires you to round a double to 1 decimal place.

The getPartInformation() method

While most methods such as the constructors and destructor for this class can be derived from existing Part classes and don't require any consideration of deep copies and dynamic memory allocation, the format for the getPartInformation() method is unclear based on the specifications. An example of how to format the std:string which is returned from this method can be seen below.

Input:

CPU cpu("Intel Core i7-3770K Ivy Bridge Quad-Core", 3.50, "LGA 1155")

Expected output:

CPU: Intel Core i7-3770K Ivy Bridge Quad-Core, Clock Speed: 3.5 Ghz, Socket: LGA 1155

Note

It is important that your formatting matches the above layout. If it does not you will fail a number of tests later in the tutorial.

Hint

If you are unsure how to format the double look into the documentation on std::stringstream covered in last week's tutorial.

Implementing the HardDriveSet

The HardDriveSet class is a wrapper class which contains the set of all Hard Drives (HardDrive) stored within the Computer class. By encapsulating the hard drives in this way we allow for queries such as the number of hard drives to be made without the Computer class requiring any additional understanding of it Parts.

One thing you will notice when comparing the HardDriveSet to either the RamSet or GraphicsCardSet classes is that the list of HardDrives are stored within a HardDrive** object. This is required to allow the storage of classes which inherit from HardDrive. This style of programming is often discouraged, with a preference for using the STL containers often seen as the prefered choice. This tutorial is intentionally designed using double pointer arrays to give you practice managing memory on the heap for yourself.

The copy constructor

To successfully achieve the allocation of memory in this exercise it is important that you carefully construct your copy constructor. This method will get called each time you transfer ownership of the HardDriveSet. It is therefore important that you implement your copy constructor to perform a deep copy, as the default behaviour is to only perform a shallow copy as discussed in the lectures. If you're still unsure about the difference between deep and shallow copies perhaps this stackoverflow post will help.

Destroying a dynamically allocated array of dynamically allocated objects

When destroying a dynamically allocated array of dynamically allocated objects (a double pointer array) you are required to not only delete the array using the delete[] notation but also need to delete each item in the array. It is important, however, that you delete the individual items first because otherwise you will no longer be able to access them.

Warning!

The implementation of this class requires you to allocate and manage a number of pointers. Make sure in your implementation that you are handling this memory appropriately to ensure that you do not have any memory leaks or segmentation faults.

Similar to the CPU class, the required functionality for the HardDriveSet is defined in its header file. As with the CPU class, the required format of the getPartInformation() method is unclear. Therefore, use the following example to assist in its implementation.

Input:

HardDrive** hardDrives = new HardDrive*[3];
hardDrives[0] = new SolidStateDrive("SAMSUNG 850 EVO-Series MZ-75E1T0B/AM", 1000);
hardDrives[1] = new HardDiscDrive("WD BLACK SERIES WD2003FZEX", 2000, 7200);
hardDrives[2] = new HardDiscDrive("WD BLACK SERIES WD2003FZEX", 2000, 7200);

Drive 1: SAMSUNG 850 EVO-Series MZ-75E1T0B/AM, Size: 1000 GB
Drive 2: WD BLACK SERIES WD2003FZEX, Size: 2000 GB, 7200 RPM

Implementing the Computer Class

The last compodent to implement is the Computer class. This class acts as a wrapper from both the required and additional parts. Both of which are stored within Part** objects.

The challenge for you in this section is to complete this class without adding any additional member variables. That is, all data is stored within the two Part** objects.

To achieve this you will need to think carefully about how you allocate and delete memory and be particularly careful how you write your destructor.

Note

Remember when you are dealing with a pointer to a pointer that you need to delete each element from the array, then deallocate the array using delete[]. The challege in this class is knowing which elements in the array have been dynamically allocated and which have not.

While destroying memory is always important, using abstraction layers when creating data structures is also an important part of the Builder design pattern. To help with this you have been provided with a helper method for the construction of a number of Part classes. This library is called the PartAllocator.

Info

It is strongly recommended that you only read the header file for this class. You have not yet been introduced to the dynamic_cast method in either the lectures or the labs. You will be introduced to it along with a number of new casting techniques in Tutorial 9. Until then it is recommended that you treat PartAllocator as a black box.

Creating a set of Computer Builders

Now that you have created the Product class and its associated Parts we can create a set of Builder classes. All classes will inherit from the following base class.

#ifndef COMPUTERBUILDER_H
#define COMPUTERBUILDER_H

#include "Computer.h"

class ComputerBuilder
{
public:

    /**
     * @brief ComputerBuilder default constructor
     */
    ComputerBuilder();

    /**
     * @brief ~ComputerBuilder destructor
     */
    virtual ~ComputerBuilder();

    /**
     * @brief createComputer executes the algorithm to assemble a computer
     */
    virtual void createComputer();

    /**
     * @brief getResult retrives the constructed computer
     * this should only be called after running createComputer()
     * @return the constructed computer
     */
    virtual Computer* getResult() const;

protected:

    /**
     * @brief addCPU add a CPU to the computer
     */
    virtual void addCPU() = 0;

    /**
     * @brief addMotherboard add a Motherboard to the computer
     */
    virtual void addMotherboard() = 0;

    /**
     * @brief addRAM add Ram to the computer
     */
    virtual void addRAM() = 0;

    /**
     * @brief addHardDrive add Hard Drives to the computer
     */
    virtual void addHardDrive() = 0;

    /**
     * @brief addGraphicsCard add Graphics Cards to the computer
     */
    virtual void addGraphicsCard() = 0;

    /**
     * @brief addPowerSupply add a Power Supply Unit to the computer
     */
    virtual void addPowerSupply() = 0;

    /**
     * @brief addCase add a Case to the computer
     */
    virtual void addCase() = 0;

    /**
     * @brief addAdditionalParts add any additional parts to the computer
     */
    virtual void addAdditionalParts() = 0;

    Computer* m_computer;
};

#endif // COMPUTERBUILDER_H

What is important to note in this interface is that there are only two methods which will be accessable to the Director (the class which will use the builders). They are:

  • createComputer(): this method will create an instance of a computer based on a Builder classes specific design.
  • getResult(): this method will transfer ownership of the Computer class to the calling class.

The remaining methods within the Builder class define the beahviour all Builders must have but may not be accessed by the Director class(es).

The implementation of Builder classes requires limited coding (consistent with the design pattern), assuming that the Product classes have been implemented correctly. The Director

The final element in the Builder design pattern is the Director. The Director is the class which takes ownership of the Product after the Builder creates it. You do not need to build a Director class in today's exercise (your main.cpp is currently acting as a pseudo Director) but it is important that you are aware of the important distinctoins between the three compodents that make up the Builder design pattern.

Extension: Inbuilt Validation

If you finish the lab early or are interested in extending your Builder(s), one area which was not explored in today's lab is to validate that the Product is correct. For example the motherboard specifies the number of graphics cards which are permitted but this is not enforced. As a first step, count the number of rules that could be enforced based on the required Parts.

Using this list add a validation step to the building process so that an exception is thrown if the computer built breaks any of the rules which you have defined. If you are unsure how exceptions work in C++ read more about the C++ std::exception class here.
